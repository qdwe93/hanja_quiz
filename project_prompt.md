# 초등 한자 학습 웹앱 프로젝트 실행 프롬프트

## 역할

당신은 제품 기획, 데이터 정제, UX/UI 설계, 프런트엔드 개발, 테스트, Git/GitHub 설정을 끝까지 수행하는 시니어 프로덕트 엔지니어다. 아래 요구사항을 바탕으로 초등학교 6학년이 사용할 수 있는 한국어 한자 학습 웹앱을 완성하라.

단순히 계획이나 문서만 작성하고 멈추지 말고, 문서화 → 구현 → 테스트 → 저장소 정리까지 순서대로 진행하라. 작업 중 합리적으로 결정할 수 있는 세부 사항은 스스로 결정하되, 중요한 가정과 결정 근거는 관련 문서에 기록하라.

## 프로젝트 목표

전국한자능력검정시험 배정한자 가운데 **7급, 준6급, 6급에 해당하는 총 200자**를 학습할 수 있는 반응형 웹앱을 만든다. 대상 사용자는 초등학교 6학년이며, 핵심 학습 방식은 다음 두 가지다.

1. 한자 카드와 해당 한글 음훈 카드를 뒤집어 짝을 맞추는 **짝맞추기**
2. 제시된 한자를 보고 네 개의 한글 음훈 중 하나를 고르는 **4지선다 퀴즈**

한자만 보여 주는 일반 플래시카드 기능은 핵심 범위에 포함하지 않는다.

## 필수 작업 원칙

1. **데이터를 추측하거나 임의로 만들지 마라.** 원문 페이지에서 한자와 한글 음훈을 직접 취합하고 검증하라.
2. 대상은 8급부터 6급까지의 누적 범위 중 **8급 50자와 준7급 50자를 제외한 200자**다.
3. 등급별 예상 수량은 `7급 50자 + 준6급 50자 + 6급 100자 = 200자`다. 원문의 표기나 구조가 예상과 다르면 임의로 수를 맞추지 말고 원문을 다시 확인한 뒤 차이를 문서화하라.
4. 같은 한자를 중복 저장하지 말고, 이체자·복수 음훈·동음 또는 유사 뜻 때문에 정답이 모호해지는 경우를 데이터 규칙과 퀴즈 생성 규칙으로 해결하라.
5. 앱의 모든 사용자 노출 문구는 자연스러운 한국어로 작성하라.
6. 초등학생이 별도 설명 없이 사용할 수 있을 만큼 단순하게 만들되, 유아용처럼 지나치게 장식적이거나 상업용 카탈로그처럼 보이지 않게 하라.
7. 접근성, 모바일 사용성, 키보드 조작, 명확한 정답 피드백을 부가 기능이 아닌 완료 조건으로 취급하라.
8. 필요 이상으로 백엔드, 회원가입, 관리자 기능을 추가하지 마라. MVP는 정적 배포가 가능한 클라이언트 중심 웹앱으로 구현하라.

## 한자 데이터 요구사항

### 수집 및 정규화

원문에서 대상 한자 200자의 한자, 음, 훈, 원문 음훈 표기, 등급을 취합하라. 최소 데이터 구조는 다음 의미를 포함해야 한다.

| 필드 | 설명 |
|---|---|
| `id` | 앱 내부에서 안정적으로 사용하는 고유 식별자 |
| `hanja` | 한 글자의 한자 |
| `eum` | 한글 음 |
| `hun` | 한글 훈 또는 뜻 |
| `eumhun` | 화면에 표시할 정규화된 음훈. 예: `하늘 천` |
| `grade` | `7급`, `준6급`, `6급` 중 하나 |
| `source` | 출처와 검증 근거를 추적할 수 있는 값 |

- 원문의 표기 순서를 보존한 원본 값과 앱 표시용 정규화 값을 구분하라.
- 복수 음 또는 복수 훈은 원문을 손실 없이 보존하고, 학습 화면에서 어떻게 표시할지 명시적인 규칙을 정하라.
- 데이터 파일의 형식은 프로젝트 기술 스택에 맞게 JSON 또는 타입이 지정된 TypeScript 모듈 중 유지보수하기 쉬운 방식을 선택하라.
- 수집 과정, 정규화 규칙, 예외 사례, 출처, 최종 검증 결과를 `docs/DATA_SPEC.md`에 기록하라.

### 자동 검증

빌드나 테스트에서 실행할 수 있는 데이터 검증을 작성하라. 적어도 다음을 검사해야 한다.

- 전체 항목 수가 정확히 200개인지
- 등급별 항목 수가 원문과 일치하는지
- `hanja`와 `id`가 중복되지 않는지
- 필수 필드가 비어 있지 않은지
- 허용되지 않은 등급 값이 없는지
- 화면 표시용 음훈이 비어 있거나 깨진 글자를 포함하지 않는지
- 4지선다 보기 생성 시 중복 보기나 복수 정답 가능성이 생기지 않는지

## 제품 기능 요구사항

### 1. 시작 화면

- 앱의 목적과 두 학습 모드를 한눈에 이해할 수 있게 한다.
- `7급`, `준6급`, `6급`, `전체` 중 학습 범위를 고를 수 있게 한다.
- 선택한 범위의 글자 수를 보여 준다.
- 짝맞추기와 4지선다 퀴즈로 바로 이동할 수 있게 한다.
- 최근 학습 기록이 있으면 브라우저 로컬 저장소를 이용해 간단한 진행 현황을 보여 준다.

### 2. 카드 짝맞추기

- 한 쌍은 **한자 카드 1장 + 그 한자의 한글 음훈 카드 1장**으로 구성한다.
- 모든 카드는 처음에 뒷면으로 시작하며, 한 번에 최대 두 장만 공개한다.
- 정답인 두 카드는 맞춘 상태로 유지하고, 오답이면 사용자가 확인할 짧은 시간을 준 뒤 다시 뒤집는다.
- 라운드마다 카드 순서를 무작위로 섞는다.
- 한 화면에 과도하게 많은 카드를 배치하지 말고, 화면 크기와 난이도를 고려해 적절한 기본 쌍 수를 정한다.
- 현재 맞춘 쌍, 전체 쌍, 시도 횟수 또는 진행률을 명확히 표시한다.
- 라운드 완료 화면에서 다시 하기, 같은 범위의 새 문제, 다른 학습 방식으로 이동하기를 제공한다.
- 빠른 연속 클릭, 이미 맞춘 카드 재선택, 세 번째 카드 선택 등 상태 전이 오류를 방지한다.

### 3. 4지선다 퀴즈

- 문제에는 한자 한 글자를 크게 제시한다.
- 보기에는 서로 다른 한글 음훈 네 개를 제시하고 정확히 하나만 정답이 되게 한다.
- 정답 위치는 문제마다 무작위로 배치한다.
- 오답 보기는 현재 선택한 등급 범위에서 우선 생성하되, 동일한 표시 문자열이나 의미상 복수 정답이 될 수 있는 보기는 제외한다.
- 답을 고르면 정답과 오답을 색상 외에도 아이콘과 문구로 구분하고, 해당 한자의 음과 훈을 다시 보여 준다.
- 한 문제에 중복 응답하지 못하게 하며, 명시적인 `다음 문제` 동작을 제공한다.
- 한 세트 안에서는 가능한 한 같은 문제를 반복하지 않는다.
- 현재 문제 번호, 전체 문제 수, 정답 수 또는 진행률을 보여 준다.
- 종료 화면에는 점수, 정답률, 틀린 글자 복습 목록, 다시 풀기를 제공한다.

### 4. 공통 학습 경험

- 등급과 학습 모드를 쉽게 다시 선택할 수 있어야 한다.
- 새로고침 후에도 최소한 최근 선택, 완료 횟수, 퀴즈 결과 정도는 로컬 저장소에서 복원한다.
- 저장된 데이터가 손상되거나 스키마가 바뀌어도 앱이 흰 화면으로 멈추지 않도록 안전한 기본값을 사용한다.
- 소리와 과한 애니메이션은 필수가 아니다. 모션을 사용한다면 짧고 절제되게 적용하고 `prefers-reduced-motion`을 존중한다.
- 로딩, 빈 데이터, 오류, 라운드 완료 상태를 각각 설계한다.

## 디자인 적용 지침

`design.md`의 **컬러 토큰, 타이포그래피 계층, 8px 간격 체계, 카드/버튼의 두 단계 모서리 체계, 절제된 그림자, 반응형 구조, 컴포넌트 명세 형식**을 디자인 시스템의 기반으로 사용하라.

다만 원본 문서의 HP 브랜드와 제품 판매 맥락을 그대로 복제하지 말고, 한자 학습 앱에 맞게 다음과 같이 재해석하라.

- HP 로고, 제품 카탈로그, 가격표, 구매 CTA, 할인 태그, 고객 후기 등 앱과 무관한 명칭과 지침은 제거하거나 학습 기능에 맞게 바꾼다.
- 기존의 흰색·옅은 회색·짙은 잉크색 표면과 Electric Blue 중심 색상 체계는 유지한다.
- 파란색은 주요 행동, 선택 상태, 진행 표시 등 중요한 곳에만 제한적으로 사용한다.
- 카드에는 16px 안팎의 부드러운 모서리, 버튼과 입력 요소에는 4px 안팎의 비교적 선명한 모서리를 유지한다.
- 장식 모티프가 필요하면 기존의 각진 셰브론을 한자의 획이나 책갈피를 연상시키는 절제된 요소로 재해석하되, 학습을 방해하는 배경 장식으로 남용하지 않는다.
- 독점 서체인 Forma DJR Micro를 필수 의존성으로 사용하지 않는다. 한글과 한자를 모두 명확히 표시하는 공개 웹 폰트 또는 안정적인 시스템 폰트 스택을 선택하고 이유를 기록한다.
- 본문과 UI는 한국어 가독성을 우선하고, 한자 문제 글자는 화면 크기에 맞춰 충분히 크게 표시한다.
- 색상만으로 맞음/틀림을 전달하지 말고 텍스트, 아이콘, 테두리 상태를 함께 사용한다.
- 모바일 최소 터치 영역은 44×44px로 하고, 작은 화면에서도 카드와 보기 문구가 잘리지 않게 한다.

기존 `design.md`는 참고 원본으로 보존하고, 실제 앱에 맞게 갱신한 디자인 시스템 문서를 `docs/DESIGN_SYSTEM.md`에 작성하라. 토큰 이름을 변경한다면 이전 토큰과 새 토큰의 대응 관계도 남겨라.

## 기술 구현 지침

- 저장소가 비어 있다면 **React + TypeScript + Vite** 기반의 정적 SPA를 기본 선택으로 사용하라. 시작 시점의 안정적인 버전과 호환성을 확인하고 정확한 버전은 잠금 파일에 고정하라.
- 스타일은 CSS 사용자 정의 속성 또는 일관된 토큰 계층으로 구현해 `docs/DESIGN_SYSTEM.md`와 코드가 어긋나지 않게 하라.
- 게임 규칙과 문제 생성 로직을 UI 컴포넌트에서 분리해 순수 함수로 테스트할 수 있게 한다.
- 데이터 타입, 게임 상태, 로컬 저장소 스키마를 명시적으로 정의한다.
- 라우팅을 사용한다면 새로고침과 정적 호스팅 환경을 고려한다.
- 전역 상태 라이브러리는 실제 복잡성이 필요할 때만 도입하고, 불필요한 의존성을 추가하지 않는다.
- 린트, 포맷, 단위 테스트, 프로덕션 빌드 명령을 `package.json`에 제공한다.
- 핵심 로직은 단위 테스트로, 주요 사용자 흐름은 컴포넌트 또는 브라우저 테스트로 검증한다.

## 필수 문서 산출물

개발 전에 초안을 작성하고, 구현 결과와 달라진 부분은 개발 중 계속 갱신하라.

1. `README.md`
   - 제품 소개, 핵심 기능, 기술 스택, 설치/실행/테스트/빌드 방법
   - 데이터 출처와 범위
   - 프로젝트 구조
2. `docs/PRD.md`
   - 문제 정의, 대상 사용자, 목표와 비목표, 사용자 흐름, 기능 요구사항, 성공 기준, 수용 기준
3. `docs/TRD.md`
   - 아키텍처, 데이터 모델, 상태 관리, 라우팅, 저장 전략, 테스트 전략, 접근성, 성능 및 보안 고려사항
4. `docs/IMPLEMENTATION_PLAN.md`
   - 단계별 구현 순서, 의존 관계, 검증 방법, 위험 요소와 대응책
5. `docs/TASKS.md`
   - 체크박스 형태의 작은 작업 단위, 우선순위, 선행 조건, 완료 조건
6. `docs/DATA_SPEC.md`
   - 데이터 출처, 수집 방법, 스키마, 정규화 규칙, 예외, 수량 및 중복 검증 결과
7. `docs/DESIGN_SYSTEM.md`
   - 앱에 맞게 바꾼 토큰, 레이아웃, 컴포넌트, 상태, 반응형 및 접근성 규칙

문서끼리 같은 내용을 불필요하게 반복하지 말고 상호 링크를 사용하라. 모든 문서는 한국어로 작성한다.

## 권장 진행 순서

### 1단계: 현황 조사

- 로컬 파일, Git 상태, 원격 저장소 상태, 사용 가능한 런타임을 확인한다.
- `startPrompt.txt`와 `design.md`를 읽고 요구사항과 디자인 전환 항목을 요약한다.
- 원문 배정한자 페이지의 구조와 대상 등급을 확인한다.

### 2단계: 기획 및 기술 문서

- PRD, TRD, 구현계획서, Task 정의서를 작성한다.
- 데이터 스키마와 디자인 시스템을 확정한다.
- 불명확한 사항은 MVP에 가장 단순하고 안전한 가정을 적용하고 문서에 남긴다.

### 3단계: 저장소 및 앱 기반 구성

- 프로젝트를 초기화하고 기본 디렉터리, 품질 도구, 테스트 환경을 구성한다.
- 디자인 토큰과 공통 레이아웃, 공통 상태 컴포넌트를 먼저 구현한다.
- 빈 저장소에 필요한 `.gitignore`와 기본 설정 파일을 추가한다. 사용자가 선택하지 않은 라이선스는 임의로 부여하지 않는다.

### 4단계: 데이터 구축과 검증

- 200자의 데이터를 수집·정규화한다.
- 자동 검증과 데이터 테스트를 실행하고 결과를 문서화한다.
- 데이터가 검증되기 전에는 퀴즈의 완성 여부를 선언하지 않는다.

### 5단계: 기능 구현

- 시작 화면과 등급 선택
- 짝맞추기 상태 로직과 UI
- 4지선다 문제 생성 로직과 UI
- 결과 및 복습 화면
- 로컬 진행 기록
- 오류·빈 상태·반응형·접근성 처리

### 6단계: 품질 검증

- 린트, 타입 검사, 단위 테스트, 사용자 흐름 테스트, 프로덕션 빌드를 모두 실행한다.
- 모바일과 데스크톱 화면을 실제 브라우저에서 확인한다.
- 한자와 한글 폰트 표시, 긴 음훈 줄바꿈, 키보드 조작, 포커스 표시, 색상 대비를 점검한다.
- 발견한 문제를 수정한 후 검증 명령을 다시 실행한다.

### 7단계: GitHub 정리

- 현재 디렉터리를 Git 저장소로 안전하게 초기화하거나 기존 저장소를 사용한다.
- 기본 브랜치는 `main`으로 맞추고, `origin`이 없다면 `https://github.com/qdwe93/hanja_quiz.git`을 연결한다.
- 기존 원격 커밋이 있는지 먼저 확인하고, 원격 기록을 덮어쓰거나 강제 푸시하지 않는다.
- 의미 있는 작업 단위로 커밋을 나누고 명확한 커밋 메시지를 작성한다.
- GitHub Actions에서 최소한 설치, 린트, 타입 검사, 테스트, 빌드를 검증하도록 CI를 구성한다.
- 모든 검증이 성공하면 `main` 브랜치를 원격 저장소에 푸시한다.

## 완료 기준

다음 조건을 모두 충족해야 작업 완료로 간주한다.

- 검증된 대상 한자 데이터가 정확히 200자다.
- 7급, 준6급, 6급 및 전체 범위 선택이 실제 문제 데이터에 반영된다.
- 짝맞추기에서 한자와 한글 음훈의 올바른 쌍을 끝까지 맞출 수 있다.
- 4지선다에서 항상 보기 네 개와 정답 한 개가 제공되며 모호하거나 중복된 보기가 없다.
- 점수, 진행 상태, 완료 및 재시작 흐름이 정상 작동한다.
- 모바일과 데스크톱에서 레이아웃이 깨지지 않는다.
- 키보드만으로 핵심 기능을 사용할 수 있고 포커스가 명확하다.
- 새로고침과 잘못된 로컬 저장 데이터에서도 앱이 복구된다.
- 필수 문서 일곱 종류가 실제 구현과 일치한다.
- 린트, 타입 검사, 테스트, 프로덕션 빌드가 모두 성공한다.
- GitHub 저장소 설정과 CI가 완료되고, 안전하게 원격 저장소에 반영된다.
